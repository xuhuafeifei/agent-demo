# 架构设计（模型初始化与获取）

本文档只覆盖从配置解析到拿到 `model` 对象这一段，不涉及后续 LLM 请求、流式回包和前端渲染。

## 总览

模型初始化拆成 3 层：

1. `src/agent/agent-path.ts`
2. `src/agent/model-config.ts`
3. `src/agent/pi-embedded-runner/model.ts`

目标是把“配置来源复杂、供应商多样、模型兼容差异”收敛成一个稳定接口：

- 输入：`provider + modelId +（可选）配置`
- 输出：`{ model?: Model, error?: string }`

## 1. `src/agent/agent-path.ts`

职责：统一模型相关持久化目录，避免各处手写路径。

### 核心接口

- `resolveAgentDir(): string`

### 设计规则

- 支持环境变量覆盖（便于测试、多环境部署）。
- 无覆盖时使用默认目录（例如 `~/.fgbg/agent`）。
- 返回值必须是绝对路径，供 `models.json`、`auth`、运行时缓存复用。

### 关键点

实现里会统一解析 agent 目录，并把该目录传给模型发现与认证存储，保证：

- 同一个会话上下文下，模型配置和认证来源一致。
- CLI / 服务端 / 测试场景使用同一目录解析逻辑。

## 2. `src/agent/model-config.ts`

职责：构建“可发现模型”的供应商配置，并产出最终 `models.json`（或等价内存结构）。

这层不直接返回 `model` 实例，它负责“准备好可被发现的模型定义”。

### 2.1 供应商构建（Provider Builders）

示例：

- `buildMinimaxProvider()`
- `buildMoonshotProvider()`
- `buildKimiCodeProvider()`
- `buildQwenPortalProvider()`
- `buildXiaomiProvider()`
- `buildOllamaProvider()`

每个 Provider 至少包含：

- `baseUrl`
- `api`（如 `openai-completions`、`anthropic-messages`）
- `models[]`（每个模型的 `id/name/contextWindow/maxTokens/cost/...`）
- `apiKey`（可来自环境变量、配置文件、profile）

### 2.2 隐式供应商解析（Implicit Providers）

推荐自动检测可用供应商：

- 从环境变量读取（如 `MINIMAX_API_KEY`）。
- 从本地 profile/credential store 读取。
- 对特殊 provider 做定制逻辑（如 OAuth 占位、AWS SDK 模式）。

输出为 `Record<string, ProviderConfig>`。

当前项目实现中，`resolveImplicitProviders` 已支持：

- `minimax`
- `moonshot`
- `kimi-code`
- `qwen-portal`
- `xiaomi`
- `ollama`（本地 provider，默认加入；有 key 则附带 key）

### 2.3 显式配置合并（Explicit + Implicit）

显式配置来自项目配置文件；隐式配置来自环境与本地认证。

合并策略建议：

- 默认 `merge` 模式。
- 显式配置优先覆盖隐式配置的同名字段。
- `models[]` 以 `id` 去重合并：显式优先，隐式补齐。

### 2.4 规范化（Normalization）

对最终 provider 配置做一次规范化：

- 修正 provider id 别名（如 `z.ai -> zai`、`qwen -> qwen-portal`）。
- 修正错误的 apiKey 配置形态（例如 `"${MINIMAX_API_KEY}"` 归一化为变量名/值）。
- 为缺失 apiKey 的 provider 尝试二次注入（env/profile）。
- 统一模型 id 兼容映射（例如某些 provider 的 preview id）。

### 2.5 ensureModelJson()

职责：把最终可用 provider 配置落盘到 `models.json`。

建议行为：

- 如果没有任何可用 provider，不写文件，返回 `wrote: false`。
- 写入前做内容比对，内容一致则跳过写入。
- 写文件权限收敛（目录 `700`，文件 `600`，按需）。

返回：

- `{ agentDir: string, wrote: boolean }`

## 3. `src/agent/pi-embedded-runner/model.ts`

职责：封装 pi-core/pi-coding-agent 的模型发现与选型，输出可直接用于 Agent 的 `model` 对象。

## 3.1 核心接口

- `resolveModel(provider: string, modelId: string, agentDir?: string, cfg?: Config)`

返回：

- `model?: Model`
- `error?: string`
- （可选）`authStorage`、`modelRegistry` 用于调试

## 3.2 获取 model 的标准流程

1. 解析 `agentDir`。
2. 创建认证存储（`discoverAuthStorage(agentDir)`）。
3. 创建模型注册表（`discoverModels(authStorage, agentDir)`）。
4. 先从 registry 查找：`modelRegistry.find(provider, modelId)`。
5. 命中则返回标准化后的 model。
6. 未命中则尝试 inline providers（来自配置中的 providers.models）。
7. 再未命中则走受控 fallback（仅当 provider 已配置或 mock 场景）。
8. 全部失败返回 `error: Unknown model: provider/modelId`。

## 3.3 为什么不是 if-else 堆满 server

这里采用“集中式模型解析器”思路，而不是在调用方写很多 `if (provider===...)`：

- 调用方只依赖 `resolveModel(...)` 一个入口。
- provider 扩展只改模型层，不污染业务层。
- 错误可观测性更好（统一 error 格式和调试信息）。

这本质是“策略注册 + 统一入口”，接近策略模式的工程化落地。

## 4. 配置优先级与读取路径

当前实现增加了全局用户配置文件：

- 默认路径：`~/.fgbg/fgbg.json`
- 可通过环境变量覆盖：`FGBG_CONFIG_PATH`

读取顺序（同字段覆盖优先级）为：

1. 环境变量（最高优先，例如 `MINIMAX_API_KEY`）
2. 项目配置 `config/model.json`
3. 全局配置 `fgbg.json`
4. 代码内默认值（兜底）

说明：`model-config.ts` 会先合并“全局 + 项目”，然后 API Key 解析时再由环境变量覆盖。

## 4.1 配置优先级（建议）

可采用以下优先级：

1. 运行时环境变量（最高优先）
2. 本地认证存储（profile/keychain/file）
3. 项目配置文件（默认值与结构约束）
4. 内置 provider 默认值（兜底）

说明：优先级是“值解析优先级”，不等同于“provider 是否可用”。可用性还受 `models[]` 是否存在、auth 是否完整、discover 是否成功影响。

## 5. 最小调用链（到拿到 model 为止）

```ts
const agentDir = resolveAgentDir();
await ensureModelJson(); // 准备 providers/models
const result = resolveModel("minimax", "MiniMax-M2.1", agentDir, cfg);

if (!result.model) {
  throw new Error(result.error ?? "model resolve failed");
}

// 到这里，拿到了可用于后续推理的 model 对象
const model = result.model;
```

## 6. 本项目裁剪建议

当前版本已从“单 provider 最小闭环”升级为“多 provider 隐式解析 + 统一解析入口”。

建议保留：

- 多 provider builder（至少保持当前 6 个）
- `resolveImplicitProviders` 的环境变量自动发现
- `ensureModelJson` 的 merge + 幂等写入
- `resolveModel` 的 registry 命中 + inline fallback

可以先不做：

- 多 provider OAuth 流程
- Bedrock/Copilot 这类特殊鉴权分支
- 复杂 alias/allowlist 规则





# 整体项目设计

本节定义重构后的目标结构，用于把当前臃肿的 `server.ts` 拆分为“Web 层 + Agent 运行层 + Pi 执行层”。

## 目标调用链

前端 -> `src/middleware/web-layer.ts` -> `src/agent/run.ts` -> `src/agent/pi-embedded-runner/attempt.ts`

## 分层职责

### 1) `src/server.ts`（应用装配层）

职责只保留：

- 初始化 Express、CORS、静态资源。
- 注册路由（把 `/api/chat` 等路由挂到 web layer）。
- 启动端口监听与基础日志。

不再承载：

- 模型选择细节。
- Agent 事件流解析。
- SSE 回包拼装细节。

### 2) `src/middleware/web-layer.ts`（Web 中间层）

职责：

- 接收 HTTP 请求并做参数校验。
- 建立 SSE 响应头、维护连接生命周期。
- 调用 `run.ts` 暴露的方法，把流式事件转发给前端。
- 将错误统一转换为 HTTP/SSE 错误格式。

建议对外接口：

- `createWebLayer()`：返回 Express Router，注册 `/api/chat`、`/api/history`、`/api/clear`。

### 3) `src/agent/run.ts`（Agent 运行编排层）

职责：

- 负责“每次对话请求”的统一流程编排。
- 管理模型发现与选型（复用 `selectModelForRuntime`、`ensureModelJson`）。
- 管理会话读写（读取历史、落盘、清理）。
- 收集并转发流式事件给上层回调（不直接依赖 Express）。

建议对外接口：

- `getReplyFromAgent(params)`：执行一次会话并流式返回事件。
- `getHistory()`：读取会话历史。
- `clearHistory()`：清理会话历史。
- `logRuntimePaths()`：打印全局配置和会话路径。

说明：

- 该层采用 FP 思路：仅导出函数，不创建长生命周期 runtime 对象。
- `getReplyFromAgent` 返回的是标准化事件流，不是 HTML/SSE 字符串。
- 该层可记录请求时间线（request start、LLM 首包、request end）。

### 4) `src/agent/pi-embedded-runner/attempt.ts`（Pi 执行层）

职责：

- 仅负责调用 pi-core/pi-agent-core 的执行逻辑。
- 把 Pi 原始事件映射成项目内部统一事件。
- 通过回调逐条上报流式增量，不关心 Web 协议。

建议对外接口：

- `createAgentRuntime(params)`：根据入参创建单次请求使用的轻量 Agent。
- `runEmbeddedPiAgent(params)`：执行一次 prompt，流式回调事件。

说明：

- `createAgentRuntime` 在每次请求时创建，不在进程全局缓存。
- 该层不处理路由、不处理 SSE、不处理 HTTP 状态码。

## 会话与模型初始化时机

在应用启动阶段（`server.ts`）：

- 打印运行路径：`logRuntimePaths()`

在请求阶段（`run.ts`）：

- 每次请求都执行一次 `selectModelForRuntime()` 选型。
- 读取会话历史后，调用 `attempt.ts` 的 `createAgentRuntime(params)` 创建轻量 Agent。
- 调用 `runEmbeddedPiAgent()` 执行并流式回调，结束后落盘会话。

会话文件落盘目录约定：

- `src/agent/session/*`

## 流式事件标准（项目内）

建议统一以下事件类型，web-layer 只做透传：

- `message_start`
- `message_update`（包含 `delta`）
- `message_end`
- `done`
- `error`

这样上层前端不需要关心 Pi 事件细节，只消费统一协议。

## 为什么这样拆

- `server.ts` 变薄：只负责装配和启动。
- 可测试性提升：`run.ts` 与 `attempt.ts` 可单测，不依赖 Express。
- 可替换性提升：后续若替换 pi-core，只需改 `attempt.ts`。
- 可观测性提升：请求耗时、首包耗时集中在运行层记录。

# 记忆系统

## 1. 模块设计

当前实现采用 `MemoryIndexManager` 作为门面层，负责组合监听、同步和检索。

### 1.1 模块划分

- `src/memory/memory.ts`
  - `MemoryIndexManager` 生命周期管理：`start/stop`
  - 监听管理：`chokidar` 监听 `MEMORY.md`、`~/.fgbg/memory/*.md`、`workspace/userinfo/*.md`
  - 事件防抖：`1.5s debounce`
  - 会话事件入口：`onMemorySourceChanged("session", sessionFile)`
  - 对外能力：`syncAll()`、`search()`

- `src/memory/indexer.ts`
  - 同步编排层：`syncMemoryByPath`、`syncAllMemorySources`
  - 状态判定：`create/rebuild/delete/skip`
  - 文件哈希比对 + 切块 + embedding + 写库调用

- `src/memory/store.ts`
  - 存储层：`node:sqlite` + `sqlite-vec`
  - 数据表初始化与事务写入
  - FTS 查询、向量查询、按 path 删除

- `src/memory/recall.ts`
  - 检索融合层：`FTS + Vec + RRF`
  - 业务加权：`MEMORY.md +10%`、`userinfo +10%`、`sessions +5%`

- `src/memory/embedding/index.ts`
  - `ensureEmbeddingProviderReady()`：补齐 `fgbg.json` 中 `agents.memorySearch` 默认配置
  - `embeddingText / batchEmbeddingText`
  - 当前实现为确定性本地向量占位（384 维），后续可替换为真实模型推理

- `src/memory/utils/*`
  - 路径、hash、切块工具函数

### 1.2 数据源与触发

- 数据源：
  - `~/.fgbg/workspace/MEMORY.md`
  - `~/.fgbg/workspace/userinfo/*.md`（顶层，参与索引）
  - `~/.fgbg/memory/*.md`
  - `sessionId.jsonl`
- **不索引**：`workspace/skills/**`（Skill 正文仅通过 `loadSkill`）；全量同步时会清理历史上误写入索引的 skills 路径。

- 触发：
  - 文件源：`chokidar add/change/unlink`
  - 会话源：在 `src/agent/run.ts` 的对话结束 `finally` 中触发 `onMemorySourceChanged("session", prepared.sessionFile)`

### 1.3 存储模型（与代码一致）

- `files(path UNIQUE, file_hash, update_time, created_time)`
- `chunks(id, source, path, chunk_content, embedding, line_start, line_end, created_at, updated_at)`
- `chunks_fts`（FTS5 virtual table）
- `chunks_vec`（`vec0` virtual table，`embedding float[384]`）

关键约束：

- `UNIQUE(path, line_start, line_end)`
- `chunks.embedding` 与 `chunks_vec.embedding` 双存（向量冗余）
- 所有 `path` 重建流程在事务中执行

### 1.4 同步策略

`syncMemoryByPath` 规则：

- 文件不存在且库中存在：`delete`
- 文件存在且 hash 不变：`skip`
- 文件不存在于库：`create`
- 文件存在且 hash 变化：`rebuild`

单文件重建顺序：

1. 删除旧 `chunks/chunks_vec/chunks_fts`（按 path）
2. 切块（保留 `line_start/line_end`）
3. 批量 embedding
4. 写入 `chunks`（含 embedding 冗余）
5. 写入 `chunks_vec`
6. 写入 `chunks_fts`
7. 更新 `files.file_hash`

### 1.5 召回融合策略

- FTS 召回：`topK=20`
- 向量召回：`topK=20`
- RRF：`score = 1/(60 + rank_fts) + 1/(60 + rank_vec)`
- 轻量加权：`MEMORY.md +10%`，`userinfo +10%`，`sessions +5%`
- 最终返回：`topN=8`（默认）

### 1.6 运行约束与现状

- 依赖 `node:sqlite`，需 Node 22 环境。
- `sqlite-vec` 通过 `loadExtension(getLoadablePath())` 加载。
- 当前 `embedding` 是占位实现；文档中提到的 `node-llama-cpp` 真实推理尚未接入。
- 定时巡检任务（每 30 分钟）与 `index_meta` 重建触发目前仅为待办，代码未实现。

### 2.2 同步编排接口

# 日志系统

实现位置：`src/logger/logger.ts`

目标：替换旧日志系统，统一“文件日志 + 控制台日志 + 子系统日志”能力。

## 配置结构（`fgbg.json`）

```json
{
  "logging": {
    "cacheTime": 300,
    "level": "info",
    "file": "/tmp/fgbg/fgbg-YYYY-MM-DD.log",
    "consoleLevel": "info",
    "consoleStyle": "pretty",
    "allowModule": ["agent", "memory"]
  }
}
```

字段说明：

- `cacheTime`：配置缓存秒数，默认 `300`（5 分钟）
- `level`：写文件最低级别
- `file`：日志文件模板路径，支持 `YYYY-MM-DD`
- `consoleLevel`：控制台最低级别
- `consoleStyle`：`pretty | common | json`
- `allowModule`：子系统日志白名单（仅作用于 subsystem logger）
  - 可写字符串：`"agent,memory"`
  - 推荐数组：`["agent", "memory"]`
  - 空数组或未配置：放行所有子系统
  - 支持 `"*"`：放行所有子系统

级别枚举：

- `trace`、`debug`、`info`、`warn`、`error`、`fatal`、`silent`

## 对外接口

- `ensureLoggingSetting(): LoggingConfig`
  - 启动阶段调用（`server bootstrap`）
  - 确保 `fgbg.json.logging` 存在且字段完整
  - 缺失字段自动写回默认值
  - 同时刷新内存缓存

- `getLogger(): Logger`
  - 全局 logger，仅写文件
  - 不带子系统名

- `getSubsystemLogger(subsystem: string): Logger`
  - 子系统 logger，仅写文件
  - 带子系统名，受 `allowModule` 过滤

- `getSubsystemConsoleLogger(subsystem: string): Logger`
  - 子系统 logger，写文件 + 控制台
  - 带子系统名，受 `allowModule` 过滤

## 缓存与读取策略

- 每次打印日志先读内存缓存
- 缓存未过期：直接使用缓存配置
- 缓存过期：重新读取 `fgbg.json` 并刷新缓存
- 读取失败时降级为默认配置，不中断主流程

## 文件切分与清理

- 单日日志文件按 5MB 自动切分：
  - 主文件：`fgbg-YYYY-MM-DD.log`
  - 超过 5MB 后续写入：`fgbg-YYYY-MM-DD-1.log`、`fgbg-YYYY-MM-DD-2.log` ...
- 自动清理 7 天前日志文件：
  - 仅清理当前日志模板家族（同模板前缀）的历史文件
  - 不影响目录下其他非本系统文件

## 输出格式

- 主 logger：`[时间] [日志级别] [日志信息]`
- 子系统 logger：`[时间] [日志级别] [子系统名称] [日志信息]`

说明：子系统名称只在 `getSubsystemLogger/getSubsystemConsoleLogger` 下输出。



# 工具

本节定义 Agent 可调用工具层（`src/agent/tool`）的目标结构与行为约束。

目标是让模型侧只面对稳定的“工具契约”，并与 `pi-core`/`pi-coding-agent` 的 `AgentTool` 接口保持一致，把路径校验、文件操作、记忆检索细节收敛在工具层内部。

## 存放目录

`src/agent/tool`

建议拆分：

- `index.ts`：统一导出工具定义与注册入口
- `types.ts`：工具输入输出、错误码类型
- `guards.ts`：路径与参数校验（工作区白名单、大小限制）
- `read.ts`：读工具实现
- `write.ts`：写工具实现
- `append.ts`：append 工具实现
- `update.ts`：update 工具实现
- `memory-search.ts`：memorySearch 工具实现（复用 `MemoryIndexManager`）
- `persist-knowledge.ts`：`persistKnowledge` 工具（按 `type` 写入 `~/.fgbg/memory`、`userinfo/`、`skills/`）

## 设计目标

- 提供基础工具：`read`、`write`、`append`、`update`、`memorySearch`，以及按配置挂载的 `persistKnowledge` 等自定义工具
- 所有文件操作默认限制在工作区内，防止越权访问
- 工具返回结构遵循 `AgentToolResult`，避免与 `pi-core` 契约偏离
- 错误可观测（明确错误码 + message），便于日志与排障
- 与现有 memory 子系统直接复用，不重复造检索逻辑

## 与 pi-core 契约对齐

`pi-core` 工具契约（简化）：

```ts

  content: Array<{ type: "text"; text: string }>;
  details: TDetails;
};
```

每个工具实现签名应为：

```ts

  toolCallId: string,
  params: ToolInput,
  signal?: AbortSignal,
  onUpdate?: (partial: AgentToolResult<ToolDetails>) => void,
): Promise<AgentToolResult<ToolDetails>>;
```

建议统一 details 结构：

```ts
type ToolDetails<T = unknown> = {
  ok: boolean;
  data?: T;
  error?: {
    code:
      | "INVALID_ARGUMENT"
      | "PATH_OUT_OF_WORKSPACE"
      | "NOT_FOUND"
      | "ALREADY_EXISTS"
      | "CONFLICT"
      | "IO_ERROR"
```

## 通用规则（所有工具）

- 所有路径先 `path.resolve` 归一化，再检查是否在 `workspace` 下
- 禁止访问 `workspace` 外路径，直接返回 `PATH_OUT_OF_WORKSPACE`
- 默认 UTF-8 文本；二进制文件不在本阶段支持范围
- 每次调用都写入工具日志（入参摘要、耗时、结果）
- 对超长输入做硬限制（例如单次写入内容上限 1MB）
- `content` 用于给模型可读反馈；结构化状态放 `details`

## 心跳核心架构概念（定稿）

### 1. Heartbeat / Trigger / Handler 分层

- Heartbeat：1s tick（可配置），仅负责触发 Trigger 扫描。
- Trigger：按 `next_run_time` 升序扫描任务表，限流（配置并发度/本轮上限），分发到 TaskHandler；不直接执行业务。
- TaskHandler：执行任务类型；两类 handler：
  1) 静态 Handler：固定逻辑（如清理 session、后续可加 memory 等）。
  2) 动态 Handler（DynamicAgentHandler）：调用 Agent，由 Agent 决定具体执行逻辑。
  每个 Handler 自行做超时与异常捕获。

### 2. 配置与缓存

- 配置源：`fgbg.json.heartbeat`，包含 `enabled`、`interval_ms`、`concurrency`、可选 `allowedScripts`。
- watch-dog 内部 5 分钟缓存配置；缓存过期再读文件，避免每 tick IO。
- 并发度配置夹紧范围 1–20，超出则夹到边界并记录告警。
- `enabled=false`：tick 仍运行但 Trigger 不分发任务（便于热开关）。

### 3. 表设计（SQLite）

- 主表：`task_schedule`
  - `id` (pk)
  - `task_name` UNIQUE（人类可读标识，防重复）
  - `task_type`（如 `execute_script` / `execute_agent` / `cleanup_logs`）
  - `payload_text`（JSON 字符串）
  - `schedule_kind`：once | cron
  - `schedule_expr`：
    - once：`runAt`（ISO 时间字符串）
    - cron：内部统一 6 段 `sec min hour dom mon dow`（对外只接受 5 段，系统补 `sec=0`）
  - `timezone`：默认 `Asia/Shanghai`
  - `status`：pending | running | done | failed | timeout
  - `attempts`：累计执行次数
  - `last_error`：最近一次失败/超时摘要
  - `create_time`，`next_run_time`, `update_time`（中国时区字符串，`+08:00`）
  - 说明：`next_run_time` 是调度缓存（derived state），用于索引与高效扫描；不是对外语义字段
  - 索引：`(status, next_run_time)`
- 明细表：`task_schedule_detail`
  - `id` (pk)
  - `task_id`
  - `start_time` / `end_time`, `create_time`, `update_time`（中国时区字符串，`+08:00`）
  - `status`：success | failed | timeout
  - `error_message`
  - `create_time`
  - `executor`（标识具体 Handler/Agent）
  - 说明：`duration` 不单存，按 end-start 计算；不再额外做“状态变更表”。

### 4. 扫描与执行流程

1. Heartbeat tick 触发 Trigger。
2. Trigger 查询：`status='pending' AND next_run_time<=now`，`ORDER BY next_run_time ASC LIMIT scan_limit`（`scan_limit` 可取并发度）。
3. 取到的任务进入“有限并发”执行池（默认并发 3，可配置；受夹紧规则）。
4. 执行前将主表标记 `running`，写 `started_at`；执行完成写 `done/failed/timeout`、`finished_at`、`attempts`、`last_error`，并插入明细表快照。
5. once 任务：执行后标记 `done/failed/timeout`；cron 任务：执行后计算下一次触发时间并写回 `next_run_time`，状态回到 `pending`。
   - cron next 计算：每次执行后直接使用 `cron-parser`，以 `now+1s` 为基准计算下一次触发时间（不做 cron 解析缓存，避免状态/内存风险）。
   - `next_run_time` 始终写回“未来时间”，确保不会在后续 tick 中反复命中同一个触发点。
6. 失败/超时回退：可选 backoff（如 min(60s, 2^attempts)），由 Trigger 写回 `next_run_time` 和递增 `attempts`。

### 5. Handler 约束与安全

- `execute_script`：仅允许 `app-path.ts` 工作目录下的 `scripts/` 子目录，拒绝包含 `..` 或绝对路径；可选 allowlist `allowedScripts`，不在列表直接标记 failed。
- `execute_agent`：支持 `timeoutMs`；并发度受 Trigger 池控制，避免同时占用过多 LLM/资源。
- 超时处理：`Promise.race(task, timeout)`；超时先发 `SIGTERM`（软 kill），再设二次超时强制 `SIGKILL`；记录 status=timeout。

### 6. 清理与内置任务

- 系统任务 `cleanup_logs`：每日执行一次，删除 `task_schedule_detail` 中 `create_time < now-7d` 的记录（可复用本框架调度）。

### 7. 任务量与堆积防护

- 单机、小规模（<100）场景：仍建议每 tick 限流，避免睡眠/时间跳变后一次性取出大量任务。
- 若当次 pending 超过 `scan_limit`，按 `next_run_time` 更早者优先，剩余的留待下个 tick。
  - 支持多种周期、一次性任务和不同 Agent 执行逻辑
- 监控：
  - 执行明细表提供完整历史记录
  - heartbeat + Trigger 可加日志记录扫描次数和任务触发情况



## 10. 新增 function tool

新增 function tool。允许大模型写入定时任务

### 10.1 任务类型收敛（单人私聊场景）

当前项目为单人私聊，不支持多人通信。为避免模型直接操控底层结构，watch-dog 任务类型收敛为两类：

- `execute_reminder`：确定性通知任务（不调用大模型）
- `execute_agent`：通用智能任务（调用一次 Agent，是否通知由任务配置决定）

不再新增 `execute_action` 等宽泛类型，避免语义歧义和输入不稳定。

### 10.2 payload 设计原则（对模型强约束，对存储动态）

存储层仍使用 `payload_text(JSON)`，但模型工具入参必须固定 schema。

原则：

1. 模型只填写“业务语义字段”（内容、时间、任务目标）。
2. 目标用户信息（如 `qqOpenid`）不出现在 `payload_text` 中；执行时由 handler 从 `fgbg.json` 的 `channels.qqbot.targetOpenid` 读取。
3. 工具内部将固定入参翻译为 `payload_text` 再落库。

说明：watch-dog 底层 payload 仍是动态 `Record<string, unknown>`，但仅 handler 侧解释，不直接暴露给模型。

### 10.3 execute_reminder（确定性提醒）

用途：如“每天 10 点提醒我喝水”。

#### createReminderTask 工具（给模型的入参）

- `content: string`（提醒内容，必填）
- `scheduleType: "cron" | "once"`
- `runAt?: string`（可解析时间字符串，`once` 必填）
- `cron?: string`（5 段：`min hour dom mon dow`，`cron` 必填；系统补 `sec=0`）
- `timezone?: string`（默认 `Asia/Shanghai`）
- `channels?: ("qq" | "web")[]`（默认 `["qq"]`）
- `taskName?: string`（可选，不传则系统自动生成）

#### 工具内部落库（由后端封装）

- `task_type = "execute_reminder"`
- `payload_text = { content, channels, timezone }`（QQ 目标 openid 不落库，执行时读 `fgbg.json`）
- `schedule_kind / schedule_expr / timezone`：由 `scheduleType + cron/runAt` 计算写入
- `next_run_time`：
  - once：`runAt`（统一为 `+08:00` 格式）
  - cron：用 `cron-parser` 计算 next（统一为 `+08:00` 格式）
- `status = "pending"`

#### handler 行为

- `executeReminderHandler` 读取 payload 内容；QQ 渠道目标 openid 仅来自 `fgbg.json`，按 channels 推送
- 不调用大模型
- 结果写入 `task_schedule_detail`

### 10.4 execute_agent（智能定时任务）

用途：

- 智能推送：如“每天 10 点帮我总结 AI 新闻并推送到手机”
- 内部维护：如“每天整理最近聊天信息并沉淀记忆（不通知用户）”

#### createAgentTask 工具（给模型的入参）

- `goal: string`（任务目标，必填）
- `scheduleType: "cron" | "once"`
- `runAt?: string`（`once` 使用，可解析时间字符串）
- `cron?: string`（`cron` 使用：5 段；系统补 `sec=0`）
- `timezone?: string`（默认 `Asia/Shanghai`）
- `title?: string`
- `taskName?: string`（可选，不传则系统自动生成）
- `notify?: boolean`（默认 `false`）
- `channels?: ("qq" | "web")[]`（`notify=true` 时生效，默认 `["qq"]`）
- `mode?: "evolve" | "analyze_then_notify"`（默认 `evolve`）

#### 工具内部落库（由后端封装）

- `task_type = "execute_agent"`
- `payload_text = { goal, mode, notify, channels, timezone }`（QQ 目标 openid 不落库，执行时读 `fgbg.json`）
- `schedule_kind / schedule_expr / timezone / next_run_time / status` 与 `execute_reminder` 同规则（`next_run_time` 统一为 `+08:00`）

#### handler 行为

`executeAgentHandler` 流程：

1. 解析 payload
2. 组装执行提示词（含当前时间、goal）
3. 调 `getReplyFromAgent` 获取 `finalText`
4. 若 `payload.notify === true`，按 payload.channels 推送（qq/web）
5. 若 `payload.notify === false`，仅执行内部动作（如记忆整理/知识沉淀）
6. 写执行明细与状态

### 10.5 通道选择策略（仅 notify=true 时）

- 默认 `channels=["qq"]`
- 仅当用户明确要求多端通知时，允许模型写入多个 channels
- 不采用“默认全通道广播”，避免重复打扰与失败放大

### 10.6 安全与校验

- 工具层必须校验时间格式（`HH:mm`）与时区
- `scheduleType` 与时间字段联动校验（`cron` 需要 `cron`，`once` 需要 `runAt`）
- `task_name` 自动生成并可附加业务前缀，防止冲突
- handler 执行失败不得阻塞调度器，应返回标准 `failed/timeout` 结果并落明细


## 11. 全局单令牌运行控制（Web + QQ 共用）

本节定义一个最小可落地方案：不区分 `web-layer` 与 `qq-layer`，全局同一时间只允许一个 Agent 任务运行。

目标：

- 避免并发执行工具导致文件冲突
- 避免并发写同一份记忆/会话数据
- 保持实现简单，先稳定再扩展

### 11.1 适用范围

- 入口层：
  - `src/middleware/web-layer.ts`
  - 后续新增 `qq-layer`（名称可定为 `src/middleware/qq-layer.ts`）
- 运行层：
  - `src/agent/run.ts` 的 `getReplyFromAgent(...)`

说明：本方案不依赖“新建 session”能力。即使每次对话都创建全新 Agent 实例，也能通过全局运行锁保证串行执行。

### 11.2 行为规则

统一规则（Web/QQ 都一致）：

1. 请求进入时先检查全局运行状态。
2. 若已有任务在跑，立即返回/发送：
   - `指令正在运行中，请稍后`
3. 若当前空闲，立即返回/发送：
   - `收到，正在处理指令`
4. 然后启动 Agent 执行。
5. 无论成功/失败/超时，都必须在 `finally` 里释放运行状态。

### 11.3 状态模型（最小）

建议新增一个轻量状态模块（例如 `src/agent/runtime-state.ts`）：

- `isRunning: boolean`
- `runningAgentId?: string`
- `startedAt?: number`

语义：

- `isRunning`：是否有任务占用全局执行槽位
- `runningAgentId`：当前任务追踪 ID（仅用于日志/排障）
- `startedAt`：开始时间（用于观测与超时保护）

说明：`agentId` 是“请求追踪ID”，不是“会话ID”。

### 11.4 接口建议（FP）

采用函数式接口，不引入 OOP：

- `tryAcquire(agentId: string): boolean`
  - 成功占锁返回 `true`
  - 已被占用返回 `false`
- `release(agentId?: string): void`
  - 释放锁（建议允许无参兜底释放）
- `getRuntimeState(): { isRunning: boolean; runningAgentId?: string; startedAt?: number }`

实现建议：模块级变量或闭包单例即可，不需要 class。

### 11.5 调用链约束

入口层伪流程（Web/QQ 同构）：

1. 生成 `agentId`（若上游已有 requestId，可直接复用）
2. `if (!tryAcquire(agentId))` -> 直接返回“正在运行中”
3. 返回“收到，正在处理指令”
4. 调用 `getReplyFromAgent(...)`
5. `finally { release(agentId) }`

要求：

- 锁的获取/释放必须在入口层完成，避免在深层函数重复控制
- 所有提前返回路径必须确保不误释放他人锁

### 11.6 错误与超时

最小要求：

- `getReplyFromAgent` 抛错时，入口层捕获并返回统一错误文案
- 无论错误类型如何，必须执行 `release(...)`

可选增强（后续再做）：

- 增加“超时强制释放”守护（按 `startedAt` 判定）
- 暴露只读状态接口用于 `/api/status` 诊断

### 11.7 取舍说明

该方案是“稳定优先”：

- 优点：实现最小、风险最低、能立即避免并发写冲突
- 缺点：吞吐受限，同一时间只能处理一个请求

后续若要提升并发，可在此基础上演进为“按作用域加锁”或“读写锁”，但当前阶段不建议提前复杂化。

## 12. Skills 管理（第一版：最简可用）

本节定义一个“先简单可跑”的 skills 管理方案，目标是：

- 启动时一次性加载 skills 元信息到内存
- 请求阶段只读内存，不扫描目录
- 通过 heartbeat 做目录变更感知，变化后全量重载
- 将 skills 的 meta 注入 system prompt

### 12.1 目录约定

默认工作区：`~/.fgbg/workspace`

```
~/.fgbg/workspace/
├── SKILL.md                  # 全局指导文件（可为空）
└── skills/
    ├── <skill-a>/
    │   ├── SKILL.md
    │   └── meta.json
    └── <skill-b>/
        ├── SKILL.md
        └── meta.json
```

`meta.json` 第一版建议字段：

- `id`: skill 唯一标识（可缺省，缺省时回退目录名）
- `name`: 展示名（可缺省，缺省时回退 `id`）
- `description`: 用途描述（可缺省，缺省为 `"N/A"`）

### 12.2 模块位置与职责

模块文件：`src/agent/skill/skill-manager.ts`

职责：

1. 模块内部维护 skills meta 的内存态（单例）
2. 对外暴露最小能力：
   - 获取 meta 信息（只读内存）
   - 加载 meta 信息（全量扫描并替换）
3. 监听 heartbeat 事件，按 hash 判断是否需要 reload
4. 提供 prompt 可用的 meta 文本（供 system prompt 注入）

### 12.3 对外接口（FP 风格）

采用函数导出，不要求 class API 暴露到业务层：

- `getSkillManager()`：返回单例 manager
- manager 能力（最小）：
  - `getMetaInfos(): SkillMetaInfo[]`
  - `loadMetaInfos(): SkillMetaInfo[]`
  - `getMetaPromptText(): string`

其中 `SkillMetaInfo` 结构建议：

```ts
type SkillMetaInfo = {
  skillId: string;
  name: string;
  description: string;
  skillDir: string;
  entryPath: string; // 第一版固定 "SKILL.md"
};
```

### 12.4 加载与重载策略

#### 启动加载

- manager 初始化时执行一次 `loadMetaInfos()`
- 全量扫描 `~/.fgbg/workspace/skills/*`
- 每个子目录读取 `meta.json + SKILL.md`
- 组装为内存数组并保存当前 hash

#### 心跳重载

- 订阅 `TOPPIC_HEART_BEAT`（现有 event-bus 事件）
- 每次心跳计算 skills hash
- hash 未变化：跳过
- hash 变化：执行 `loadMetaInfos()` 全量替换

### 12.5 hash 定义（第一版）

为保持实现简单，hash 输入可使用：

- 子目录名（排序后）
- 每个目录下 `meta.json` 内容（若存在）
- 每个目录下 `SKILL.md` 内容（若存在）

说明：第一版不做冲突检测、增量 patch、细粒度 diff，只做“变更即全量重载”。

### 12.6 与 Prompt 集成

`src/agent/run.ts` 在构建 system prompt 前读取：

- `getSkillManager().getMetaPromptText()`

并传入 `buildSystemPrompt(...)`，在提示词中新增 `Skills` 段落，例如：

```markdown
## Skills
- skill-id: skill name
  description: ...
  entry: SKILL.md
```

模型在每次请求都能看到最新（内存态）的 skills 元信息，但无需请求时扫描目录。

### 12.7 第一版明确不做

为保证架构清晰、实现最短路径，第一版暂不包含：

- skillId 冲突检测与冲突处理策略
- schema 严格校验（仅做宽松读取）
- 增量热更新（只做全量重载）
- 复杂的多源仓库（Git/Nacos/DB）适配

以上能力可在第二版演进，不影响当前接口形态。


# fgbg.json 配置管理（conf 模块，定稿）

本节用于统一 `fgbg.json` 的读取、默认值归一化与可控持久化写回策略，避免各业务模块各自实现“读配置 / 判完整 / 写回默认”的分散逻辑。

## 设计目标

- **单一入口**：所有模块通过 conf 读取/写回 `fgbg.json`，不允许业务模块直接操作文件。
- **每次请求读取**：当前流量不大，允许每次请求从磁盘读取并归一化，保证外部改动立刻生效（不引入缓存与 watcher 复杂度）。
- **不做迁移**：暂不考虑 schema 迁移；读取只做宽松校验与默认值归一化。
- **避免业务歧义**：对无明确默认值的字段（例如 `channels.qqbot.targetOpenid`），conf **不填、不写回**，由业务模块决定缺失时的行为。
- **写回白名单**：仅允许少量“无歧义字段/业务明确提供值的字段”持久化写回，避免 conf 读配置时意外修改用户文件。

## 两层配置模型

### 1) File-facing（对接磁盘文件的结构）

目标：尽量忠实表达文件内容，字段保持可选，适合读写文件与 patch。

- `FgbgFileConfig`（建议复用/对齐现有 `src/types.ts` 中的 `FgbgUserConfig`，作为 file-facing 类型）
- 字段大多为可选 `?`
- 只做最基本的结构校验（`object`、避免 `null`），避免运行时崩溃
- 对“无默认值”的字段保持 `undefined`，不补齐

### 2) Internal overview（内部系统使用的归一化结构）

目标：给内部模块提供结构稳定、默认值已补齐的配置视图，减少模块内重复判断。

- `FgbgConfig`（overview）
- 对**有明确默认值**的字段进行补齐（默认值只在 conf 中维护）
- 对**无明确默认值**的字段保持可选，例如：
  - `channels.qqbot.targetOpenid?: string`
- 子模块不再负责 schema 完整性判断，只做业务必填判断

## conf 模块形态（FP，无状态）

conf 使用 **FP（函数式）** 组织代码：

- 对外只导出函数（`getFgbgConfig()` / `patchFgbgFileConfig()` 等）
- **conf 自己不存储状态**：不维护单例对象、不做内存缓存、不保存“上次读取结果”
- 每次调用都按固定流程执行：**读盘 → parse →（可选）normalize → 返回**
- 写回同理：**读盘 → 合并 patch → 校验白名单 → 原子写盘 → 返回**

说明：在“每次请求都读取、流量不大”的前提下，这种实现最简单、行为最可预期。

## conf 模块建议目录结构

建议新增目录 `src/config/fgbg/`，形成明确边界（全部以函数导出为主，不引入 class）：

- `src/config/fgbg/file.ts`
  - 解析配置文件路径（例如 `FGBG_CONFIG_PATH` 或默认 `~/.fgbg/fgbg.json`）
  - 读文件（不存在则返回空对象）
  - 原子写入（写临时文件后 rename）
- `src/config/fgbg/defaults.ts`
  - 仅存放“无歧义默认值”
  - 不包含如 `targetOpenid` 这类需要业务提供的值
- `src/config/fgbg/normalize.ts`
  - `file-facing -> overview` 归一化逻辑（merge defaults + 宽松校验）
  - 不做迁移
- `src/config/fgbg/index.ts`
  - 对外暴露唯一 API：读取、patch、白名单持久化写回

## 对外接口（建议）

### 读取接口（每次调用读盘，无缓存）

- `getFgbgFileConfig(): FgbgFileConfig`
  - 读盘 + parse + 返回（不补默认；无状态、无缓存）
- `getFgbgConfig(): FgbgConfig`
  - 读盘 + parse + normalize(defaults) + 返回（补“无歧义默认”；无状态、无缓存）

### 写回接口（显式 + 白名单）

写回遵循两个原则：

1. **必须显式调用**才写盘（不允许“读配置就写回”）
2. **仅白名单字段允许持久化写回**

建议接口：

- `patchFgbgFileConfig(patch: DeepPartial<FgbgFileConfig>, opts: { persist: true; reason: string }): FgbgFileConfig`
  - 读盘 -> 合并 patch -> 校验允许写回的路径 -> 原子写盘 -> 返回最新 file-facing

可选提供语义化写回接口（更安全、更可读）：

- `setQqTargetOpenid(openid: string, opts: { persist: true; reason: string }): void`
  - 仅在业务侧明确拿到 openid 且需要持久化时调用

## 写回白名单策略（建议）

白名单分两类：

### A) 业务明确提供值的字段（允许写回）

- `channels.qqbot.targetOpenid`
- `channels.qqbot.accounts.*.targetOpenid`（如存在多账号）

说明：conf 不生成这些字段的默认值，只负责持久化业务传入值。

### B) 无歧义默认值字段（可选允许写回）

用于减少用户手工配置成本（但仍要求“显式触发写回”）：

- 例如 `agents.thinking`（默认值明确，偏好项）
- 例如 `channels.qqbot.enabled`（若定义明确默认）
- 例如 `meta.lastTouchedAt` / `meta.lastTouchedVersion`（如你需要记录写回来源）

## 子模块职责边界（重构后的约束）

### 子模块做什么

- 通过 `conf.getFgbgConfig()` 或 `conf.getFgbgFileConfig()` 读取配置片段
- 做**业务必填判断**（例如缺 openid 则提示用户配置，或在拿到 openid 后调用 conf 写回）
- 不负责 schema 完整性修复

### 子模块不做什么

- 不直接读写 `fgbg.json`
- 不在运行路径上“自动写回默认值”
- 不各自维护一份默认值

## 重构点清单（实施建议）

- 将所有直接调用 `getUserFgbgConfig()` / `writeFgbgUserConfig()` 的模块，替换为 conf API（避免重复写盘策略）。
- 把散落在各模块内的“默认值”下沉到 `src/config/fgbg/defaults.ts`，由 `normalize.ts` 统一归一化。
- 对 `qq-layer` 这类需要“拿到外部事实并持久化”的场景，统一走 `conf.patch...({persist:true})` 或语义化 `setQqTargetOpenid()`。
