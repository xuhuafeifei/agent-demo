# 记忆与用户偏好重构说明（评审稿）

本文档汇总近期讨论的 **记忆检索、持久化、workspace 目录与 Skill** 相关重构目标与设计约束，供评审与排期实现。  
**约定：不考虑历史兼容与数据迁移**（可不保留 `USER.md`、`persistMemory` 等旧路径与工具）。

---

## 1. 背景与问题

- **会话记忆检索**（`memorySearch`）面向已分块索引的文本；与「当前轮对话里刚说过的话」不是同一数据源，易出现噪声命中或漏掉本轮事实。
- **单文件 `USER.md`** 难以分域维护，模型追加时容易把用户档案与杂项摘要写在一起。
- `**persistMemory` 过于自由**（任意相对路径），与「哪些目录参与索引、哪些只通过专用工具加载」的边界不清晰，不利于安全校验与产品语义。

---

## 2. 目标

1. **三类持久化语义清晰**：区分「用户记忆库」「用户偏好档案」「可复用 Skill」，由调用方显式指定类型，类型决定写入目录与正文格式。
2. **检索边界清晰**：`memorySearch` **不索引、不返回** `workspace/skills/` 下正文；Skill 仅通过 `**loadSkill`**（及 prompt 中的 skill 描述列表）使用。
3. **用户偏好可检索**：`workspace/userinfo/` 下扁平 `*.md` **纳入记忆索引**，便于关键词/向量召回。
4. **废除** `persistMemory`，以 **强约束 API**（类型 + 字段校验 + 固定根目录）替代。

---

## 3. Workspace 目录约定


| 路径（相对 workspace 根） | 结构                                                   | 用途                               |
| ------------------ | ---------------------------------------------------- | -------------------------------- |
| `userinfo/`        | **扁平**：仅该目录下一层 `*.md`                                | 用户偏好、称呼、协作习惯等；正文格式见下文「Skill 样式头」 |
| `skills/`          | **树形**：`skills/<skillDir>/SKILL.md`（及 `meta.json` 等） | 可复用流程/能力；**不参与 memorySearch 索引** |
| `MEMORY.md`        | 单文件                                                  | 工作区级长期备忘（与现有设计一致，是否索引保持现状即可）     |


说明：

- 目录名采用 `**userinfo`**（不再使用 `_user` 等临时命名）。
- **不需要 `INDEX.md`**：Skill 的「列表 + 描述」由现有机制（如扫描 `meta.json`）生成 prompt；`userinfo` 的摘要可从各文件头部元信息解析后注入 prompt，或由运行时扫描生成简短列表。

---

## 4. 持久化工具设计（替代 `persistMemory`）

建议单一工具入口，**必选字段 `type`**，取值：

### 4.1 `type: "memory"`

- **写入位置**：用户记忆目录（当前实现为 `~/.fgbg/memory/`，与 workspace 分离）。
- **字段**：`fileName`（仅允许安全 basename，如 `*.md`）、`content`。
- **正文格式**：**纯 Markdown**，**不需要** Skill 样式头（无 `name`/`desc` 分隔块）。
- **行为建议**：与旧 `persistMemory` 对 memory 目录行为一致——**存在则追加，不存在则创建**（若产品希望改为覆盖，需单独拍板）。

### 4.2 `type: "skill"`

- **写入位置**：`workspace/skills/<skillDir>/`。
- **字段**：`skillDir`（合法相对段，禁止 `..` 等）、`name`、`description`、`content`（Skill 正文，可不含头，由实现统一拼头）。
- **正文格式**：与 **Skill 文档约定** 一致，例如文件开头使用固定分隔块携带元信息（具体分隔符以实现为准，需与解析器、`loadSkill` 预期一致）。
- **元数据**：建议同时维护 `**meta.json` 的 `name` / `description`**，以便与现有 `**getSkillManager().getMetaPromptText()`** 一致，prompt 中只出现描述，正文靠 `loadSkill` 加载。
- **写入后**：可触发 Skill 元信息 **reload**（如心跳或立即 `loadMetaInfos`），避免描述滞后。

### 4.3 `type: "userinfo"`

- **写入位置**：`workspace/userinfo/<fileName>.md`（扁平，一层）。
- **字段**：`fileName`（basename）、`name`、`description`、`content`。
- **正文格式**：与 **Skill 相同头部约定**（`name` / `desc` 等），便于从文件解析摘要并注入「User / 用户偏好」类 prompt 段落。
- **索引**：`userinfo/*.md` **参与 memorySearch**（见下节）。

### 4.4 安全与校验（实现必做）

- 所有路径必须经过 **basename / 白名单根目录** 校验，禁止 `..`、绝对路径、越界到 `skills` 外写入 skill 等。
- `content` 保留合理大小上限（沿用现有工具链中的文本上限思路即可）。

---

## 5. 记忆索引（`memorySearch`）范围

**应索引（保持或新增）**

- 工作区 `MEMORY.md`（若当前已在索引中则维持）。
- `~/.fgbg/memory/*.md`（用户记忆库，`type: "memory"` 写入目标）。
- **新增**：`workspace/userinfo/*.md`（仅顶层，与「扁平」约定一致）。
- 会话 `*.jsonl`（若当前已在索引中则维持）。

**明确不索引**

- `**workspace/skills/`** 下任意 Markdown（尤其 `SKILL.md` 正文）**，避免与「仅描述进 prompt、正文用 `loadSkill`」冲突。

**实现注意**

- 需在 **全量同步** 与 **文件监听** 中纳入 `userinfo/*.md`

**检索侧**

- 可在 RRF 加权中为 `**userinfo` 来源** 设置与重要备忘相当的权重（具体系数实现时调参），降低被超长 session 淹没的概率。

---

## 6. Prompt 与工具分工


| 能力            | 方式                                                                                     |
| ------------- | -------------------------------------------------------------------------------------- |
| Skill 有哪些、干什么 | **System prompt「Skills」段**：来自 `meta.json` 等元数据（现有 SkillManager 思路）                     |
| Skill 完整步骤/正文 | `**loadSkill(skillDir)`**，不依赖 `memorySearch`                                           |
| 用户偏好摘要        | **System prompt「User」或独立小节**：由 `userinfo/*.md` 解析出的 `name`/`description`（或等价字段），不必全文灌入 |
| 用户偏好全文 / 模糊回忆 | `**memorySearch` + `read`**                                                            |

## 9. 小结

本次重构的核心是：**用显式 `type` 绑定目录与格式**，把 **可检索的用户偏好（userinfo）** 与 **不可检索正文的 Skill（skills）** 分开，并 **废除宽口径的 `persistMemory`**。  
实现阶段以本文第 7 节为检查列表即可逐项落地。
