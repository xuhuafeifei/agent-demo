# 工具安全性重构技术方案

## 📋 1. 项目背景与范围

### 1.1 项目概述

- **项目名称**：agent-demo  
- **定位**：单人维护的**日常 Agent**，附带**轻量级 coding** 能力（非 IDE 级、非专业 DevOps 场景）  
- **技术栈**：Node.js + TypeScript + @mariozechner/pi-agent-core  
- **使用场景**：Web + QQ 等多通道对话  

**范围约定（本方案默认成立）**：

| 约定 | 说明 |
|------|------|
| **单用户 / 单 workspace** | 仅一个工作区根目录；不设计多租户、不区分「组织」与「空间」 |
| **未来扩展** | 不计划支持多租户 / 多 workspace；方案按此简化 |
| **数据与迁移** | 当前为测试 demo，历史配置与数据可丢弃；**不单独写迁移方案** |

### 1.2 当前安全现状

**已有机制**：

- 路径守卫：防止路径逃逸（如 `../`）  
- 工具注册表：由配置控制可用工具  
- 工具列表：各模式只持久化一份 `tools`（不再拆 `customTools` / `innerTools`）  

**主要问题**：

1. `shellExecute` 可执行任意命令（风险最高）  
2. `read` 未区分文本与二进制，可能误读或泄露非文本内容  
3. `write` 与路径、范围缺少与模式一致的策略  
4. 缺少统一黑名单与跨平台路径语义  
5. 缺少确认、超时与资源上限（按模式分阶段落地）  

### 1.3 设计目标

- **简单**：默认模式即可用，少碰配置  
- **可预期**：同一模式下行为固定（**内置工具表**见 §2.1.4）  
- **跨平台**：路径、Shell、黑名单在 macOS / Linux / Windows 上均有明确策略（见 §2.2.3、§2.3、§2.4）  
- **敏感信息**：黑名单规则与命中细节**不向模型或终端大段回显**（见 §2.3.1）

## 🎯 2. 核心设计方案

### 2.1 模式化工具权限系统

#### 2.1.1 模式定义

```typescript
export type ToolMode = 'safety' | 'guard' | 'yolo' | 'custom';
```

**默认模式：`guard`**（配置缺省时按 `guard` 处理，与下文示例一致。）

#### 2.1.2 各模式内置工具表（重要说明）

- **`safety` / `guard` / `yolo` / `custom` 均在代码内自带一份默认工具名列表**（`MODE_TOOL_SETS`，见 §2.1.4）。  
- 用户选择的是 **模式**，不是逐工具拼装；**非 `custom` 时以模式对应列表为准**。  
- **`custom`**：在模式表上限内，再用 `toolSecurity.custom` 做细粒度开关（布尔表等）。  

#### 2.1.3 模式特性对比

| 模式 | 工具集合（内置表） | 访问范围（默认） | 确认（默认） | 适用场景 |
|------|-------------------|------------------|--------------|----------|
| **safety** | 最小集 | 仅 workspace | 无 | 日常闲聊、低风险 |
| **guard** | 扩展集，无 shell | workspace 为主，可按配置放宽, 比如用户目录 ~/ | 写/高危前确认（落地时分阶段） | **默认**；轻量写代码、任务 |
| **yolo** | 含 `shellExecute` 等 | 最宽 | 无 | 本机自动化、完全信任环境 |
| **custom** | 与表 + 用户开关组合 | 见 `custom.access` | 见 `custom.confirmation` | 需要逐项微调时 |

#### 2.1.4 模式工具集合定义（`MODE_TOOL_SETS`）

各模式只维护一份 `tools` 数组（配置与持久化不拆分子集合）。

### 2.2 受限工具专项：`read` / `write` / `shellExecute`

以下三类为**重点限制**对象；其它工具仍受模式与路径策略约束，但不展开等同细度。

#### 2.2.1 `read`：仅允许读取**文本文件**

**统一规则**：

1. 路径必须先通过 workspace / 作用域与黑名单校验（见 §2.3）。  
2. **仅当判定为文本文件时允许读取**；二进制、未知且无法排除二进制的文件 **拒绝读取**。  
3. 判定策略与 §2.3.4 一致：`TEXT_EXTENSIONS` / `BINARY_EXTENSIONS` / 文件头 sniff；**不再采用「未知扩展名默认放行」**（与「只读文本」一致，未知倾向拒绝或要求用户改扩展名）。  

**与实现的关系**：当前 `read` 使用 `utf8` 读入；加上述 gate 后，语义与「只读 text file」一致。

#### 2.2.2 `write`

1. **路径**：必须在允许的 workspace / 作用域内；黑名单与用户 `denyPaths` 与 `read` 同源检查。  
2. **内容**：仅允许写入 **UTF-8 文本**（与工具语义一致；不用于写入任意二进制）。  
3. **可选上限**：单文件写入大小、行数上限可在实现中配置，防止大对象撑爆上下文（数值按单人项目设合理默认即可）。  
4. **确认**：在 `guard` / 部分 `custom` 策略下，覆盖或写到 workspace 外路径前需确认（具体 UI 见阶段计划，本方案只定原则）。  

#### 2.2.3 `shellExecute`：白名单命令 + 跨平台策略

**原则**：

- **不允许任意 shell 字符串**；解析出「第一个可执行名」必须在白名单内。  
- **默认禁止管道、子 shell、`&&`/`||` 链**（初版可整段拒绝含 `|`、`$()`、反引子命令等，减少注入面）。  
- **跨平台**：以「**命令 basename 白名单**」为真值；各 OS 通过 `PATH` 解析可执行文件（POSIX：`command -v`；Windows：`where`），未解析到则拒绝。  
- Windows 若无 POSIX 同名工具，可用**文档化别名**（例如仅文档层允许 `dir` ↔ `ls` 的映射由实现二选一），避免把 `cmd.exe /c` 任意字符串开放出来。  

**初版推荐白名单（可按环境删减；名称均为 argv[0] 的 basename）**  

*文件与文本*：`cat`、`head`、`tail`、`wc`、`sort`、`uniq`、`grep`（若平台无 GNU grep，可用平台等价实现并固定路径）  

*路径与目录*：`pwd`、`ls`、`dirname`、`basename`（Windows 补充 `dir` 仅当映射策略明确时）  

*运行时与包管理*：`node`、`npm`、`npx`、`corepack`、`pnpm`、`yarn`  

*版本控制*：`git`（后续可对子命令再细分，初版可审计 + 限制危险子命令）  

*系统与调试信息*：`uname`、`hostname`、`date`、`whoami`、`env`（**输出需脱敏**：见下条）、`which`、`where`、`type`、`command`  

*其它安全小工具*：`echo`、`printf`、`test`、`[`、`sleep`、`true`、`false`  

**明确不在初版白名单（示例）**：`rm`、`dd`、`mkfs`、`curl`、`wget`、`ssh`、`sudo`、`chmod`（755 等）、任意解释器直接执行远程脚本等。若轻量 coding 确需，再按模式单独开「扩展白名单档」并审计。  

**`env` / 环境变量**：敏感黑名单大、易泄露密钥；**回显与日志需过滤**常见密钥模式（`API_KEY`、`SECRET`、`TOKEN` 等），与 §2.3.1 的「不大段暴露规则」一致。

### 2.3 全局安全机制

#### 2.3.1 黑名单与「大消息 / 敏感」约定

- **全局黑名单**：路径模式列表在代码或配置中维护；**命中时对模型与用户界面只返回简短原因**（例如「路径不允许访问」），**不返回完整规则表、不匹配细节的长文本**，避免被用作侧信道或刷屏。  
- **日志**：可记录命中类别（如 `deny_global`）；生产环境避免把完整用户路径 + 完整规则打满一行（单人项目仍建议脱敏）。  
- **用户自定义 `denyPaths`**：同上，错误信息短、可诊断即可。  

#### 2.3.2 全局路径模式（跨平台）

按 OS 维护多组模式，匹配前将路径 **规范化为真实路径** 再比较（`path.normalize` + 必要时 `realpath`）。下列为**结构示例**，实现时用最小 glob/前缀库统一语义。

**POSIX（macOS / Linux）示例**

```typescript
const GLOBAL_DENY_PATHS_POSIX = [
  '**/.env',
  '**/.env.*',
  '**/.secrets',
  '**/.ssh/**',
  '**/.gnupg/**',
  '**/.aws/**',
  '**/.azure/**',
  '**/.gcloud/**',
  '/etc/**',
  '/System/**',           // macOS
  '/private/**',          // macOS
  '/usr/bin/**',          // 可按需收紧；避免误伤则改为「敏感系统目录」白名单式
  '**/node_modules/**',   // 可选：降噪音/体积
];
```

**Windows 示例（示意）**

```typescript
const GLOBAL_DENY_PATHS_WIN = [
  '**/AppData/Roaming/**',           // 按需细化，避免整棵 AppData
  '**/.ssh/**',
  '**/NTUSER.DAT',
  '**/Cookies/**',
  'C:/Windows/System32/config/**',
  '**/.env',
  '**/.aws/**',
];
```

同一进程内根据 `process.platform` 选用对应表；**不要**假设只有 Linux 路径。  

#### 2.3.3 路径安全检查（与文本判定）

**核心功能**：

1. 规范化与 `..` 逃逸检测（规范化后再判断）。  
2. 在允许的作用域内（默认：单 workspace 根）。  
3. 全局 + 用户黑名单。  
4. **`read`**：通过 §2.3.4 文本判定；**`write`**：目标路径合法且策略允许创建/覆盖。  

```typescript
interface PathCheckResult {
  allowed: boolean;
  realPath: string;
  reason?: string; // 短原因，不含完整黑名单
}

async function checkPathSafety(
  inputPath: string,
  purpose: 'read' | 'write',
  config: SecurityConfig,
): Promise<PathCheckResult> {
  // 1. 规范化并禁止逃逸
  // 2. realpath（读已存在文件；写路径需单独处理「父目录存在、文件可创建」）
  // 3. 黑名单（matchesGlobal / matchesUser）—— 仅返回短 reason
  // 4. purpose === 'read' → must be text file（§2.3.4）
  return { allowed: true, realPath: '...' };
}
```

#### 2.3.4 文件类型检测（与 `read` 一致）

**策略**：

1. 扩展名在 `BINARY_EXTENSIONS` → 拒绝读。  
2. 扩展名在 `TEXT_EXTENSIONS` → 允许读（仍可做大小上限）。  
3. 无扩展名或未知：读取文件头 sniff；**疑似二进制则拒绝**；无法判断时 **倾向拒绝**（与「只读文本」一致）。  

```typescript
import path from 'node:path';

async function isTextFile(filePath: string): Promise<boolean> {
  const ext = path.extname(filePath).toLowerCase();
  if (BINARY_EXTENSIONS.has(ext)) return false;
  if (TEXT_EXTENSIONS.has(ext)) return true;
  const buffer = await readFileHeader(filePath, 512);
  if (isBinaryMagicNumber(buffer)) return false;
  return false; // 未知：不默认放行
}

const TEXT_EXTENSIONS = new Set([
  '.txt', '.md', '.json', '.yaml', '.yml', '.xml',
  '.js', '.ts', '.tsx', '.jsx', '.mjs', '.cjs',
  '.py', '.sh', '.bash', '.zsh', '.ps1',
  '.html', '.css', '.scss', '.less',
  '.c', '.cpp', '.h', '.hpp', '.java', '.go', '.rs',
]);

const BINARY_EXTENSIONS = new Set([
  '.exe', '.dll', '.so', '.dylib', '.a', '.lib',
  '.png', '.jpg', '.jpeg', '.gif', '.bmp', '.ico', '.webp',
  '.mp3', '.mp4', '.avi', '.mov', '.wav',
  '.zip', '.tar', '.gz', '.bz2', '.7z', '.rar',
  '.pdf', '.doc', '.docx', '.xls', '.xlsx',
]);
```

（扩展名集合与实现常量保持单一来源即可。）

### 2.4 沙箱与命令执行（跨平台）

#### 2.4.1 阶段划分（与跨平台）

| 阶段 | 内容 | 跨平台说明 |
|------|------|------------|
| **1** | 白名单解析、`spawn`/`execFile` 可控参数、超时、`cwd=workspace`、环境变量删减 | Windows 用可执行文件绝对路径 + 参数数组，避免 `cmd /c` 整串 |
| **2** | 审计日志、资源上限（CPU/内存/输出截断） | 使用 Node 与子进程通用能力 |
| **3（可选）** | chroot、seccomp、network namespace、cgroup | **仅 POSIX**；Windows 无对等项则跳过并文档说明 |

**不要求**单人项目在 Windows 上实现 chroot；**跨平台底线**是阶段 1 + 2。  

#### 2.4.2 命令预检（伪代码修正）

```typescript
async function preExecuteCheck(
  command: string,
  config: SandboxConfig,
): Promise<void> {
  if (command.length > 10000) throw new Error('命令过长');
  if (containsShellMetacharacters(command)) {
    throw new Error('不支持管道与链式 shell（初版）');
  }
  const argv = parseArgvSimple(command); // 或强制 JSON 数组入参
  const bin = path.basename(argv[0]!);
  if (!SHELL_ALLOWLIST.has(bin)) {
    throw new Error('命令不在允许列表中');
  }
  if (!config.network && mayAccessNetwork(argv)) {
    throw new Error('网络类命令已禁用');
  }
}
```

## 📊 3. 实现计划（单人项目粒度）

### 3.1 阶段划分

| 阶段 | 目标 | 要点 |
|------|------|------|
| **1** | 路径 + 黑名单 + **read 文本门控** | 跨平台路径表；短错误信息 |
| **2** | **write** 范围与文本写入边界；工具注册与 **guard 默认** | 与 `MODE_TOOL_SETS` 对齐 |
| **3** | **shellExecute** 白名单 + 超时 + 环境清理 + 审计 | 无管道/链式初版 |
| **4** | 确认流（Web / QQ） | 与通道特性对齐 |
| **5（可选）** | POSIX 强隔离 | 非 Windows 必需 |

### 3.2 风险（精简）

| 风险 | 缓解 |
|------|------|
| 白名单被绕过（换名、路径） | 只认 `basename` + PATH 解析结果；禁止元字符 |
| 跨平台行为差 | 分表测试路径与命令解析 |
| 黑名单日志泄露 | 短 reason、脱敏 |

## 📝 5. 总结

- **单人、单 workspace、无多租户**；方案按轻量日常 Agent + 轻量 coding 裁剪。  
- **默认 `guard`**；各模式 **内置 `MODE_TOOL_SETS`**，用户主要选模式。  
- **`read` 只读文本**；**`write`** 文本与路径受控；**`shellExecute`** 采用 **basename 白名单 + 跨平台 spawn 策略**，并附初版命令清单。  
- **黑名单与错误回显**：短消息、防敏感信息大段外泄。  
- **跨平台**：路径表分 OS；强隔离仅作 POSIX 可选项。
