# 架构设计（模型初始化与获取）

本文档只覆盖从配置解析到拿到 `model` 对象这一段，不涉及后续 LLM 请求、流式回包和前端渲染。

## 总览

模型初始化拆成 3 层：

1. `src/agent/agent-path.ts`
2. `src/agent/model-config.ts`
3. `src/agent/pi-embedded-runner/model.ts`

目标是把“配置来源复杂、供应商多样、模型兼容差异”收敛成一个稳定接口：

- 输入：`provider + modelId +（可选）配置`
- 输出：`{ model?: Model, error?: string }`

## 1. `src/agent/agent-path.ts`

职责：统一模型相关持久化目录，避免各处手写路径。

### 核心接口

- `resolveAgentDir(): string`

### 设计规则

- 支持环境变量覆盖（便于测试、多环境部署）。
- 无覆盖时使用默认目录（例如 `~/.fgbg/agent`）。
- 返回值必须是绝对路径，供 `models.json`、`auth`、运行时缓存复用。

### 关键点

实现里会统一解析 agent 目录，并把该目录传给模型发现与认证存储，保证：

- 同一个会话上下文下，模型配置和认证来源一致。
- CLI / 服务端 / 测试场景使用同一目录解析逻辑。

## 2. `src/agent/model-config.ts`

职责：构建“可发现模型”的供应商配置，并产出最终 `models.json`（或等价内存结构）。

这层不直接返回 `model` 实例，它负责“准备好可被发现的模型定义”。

### 2.1 供应商构建（Provider Builders）

示例：

- `buildMinimaxProvider()`
- `buildMoonshotProvider()`
- `buildKimiCodeProvider()`
- `buildQwenPortalProvider()`
- `buildXiaomiProvider()`
- `buildOllamaProvider()`

每个 Provider 至少包含：

- `baseUrl`
- `api`（如 `openai-completions`、`anthropic-messages`）
- `models[]`（每个模型的 `id/name/contextWindow/maxTokens/cost/...`）
- `apiKey`（可来自环境变量、配置文件、profile）

### 2.2 隐式供应商解析（Implicit Providers）

推荐自动检测可用供应商：

- 从环境变量读取（如 `MINIMAX_API_KEY`）。
- 从本地 profile/credential store 读取。
- 对特殊 provider 做定制逻辑（如 OAuth 占位、AWS SDK 模式）。

输出为 `Record<string, ProviderConfig>`。

当前项目实现中，`resolveImplicitProviders` 已支持：

- `minimax`
- `moonshot`
- `kimi-code`
- `qwen-portal`
- `xiaomi`
- `ollama`（本地 provider，默认加入；有 key 则附带 key）

### 2.3 显式配置合并（Explicit + Implicit）

显式配置来自项目配置文件；隐式配置来自环境与本地认证。

合并策略建议：

- 默认 `merge` 模式。
- 显式配置优先覆盖隐式配置的同名字段。
- `models[]` 以 `id` 去重合并：显式优先，隐式补齐。

### 2.4 规范化（Normalization）

对最终 provider 配置做一次规范化：

- 修正 provider id 别名（如 `z.ai -> zai`、`qwen -> qwen-portal`）。
- 修正错误的 apiKey 配置形态（例如 `"${MINIMAX_API_KEY}"` 归一化为变量名/值）。
- 为缺失 apiKey 的 provider 尝试二次注入（env/profile）。
- 统一模型 id 兼容映射（例如某些 provider 的 preview id）。

### 2.5 ensureModelJson()

职责：把最终可用 provider 配置落盘到 `models.json`。

建议行为：

- 如果没有任何可用 provider，不写文件，返回 `wrote: false`。
- 写入前做内容比对，内容一致则跳过写入。
- 写文件权限收敛（目录 `700`，文件 `600`，按需）。

返回：

- `{ agentDir: string, wrote: boolean }`

## 3. `src/agent/pi-embedded-runner/model.ts`

职责：封装 pi-core/pi-coding-agent 的模型发现与选型，输出可直接用于 Agent 的 `model` 对象。

## 3.1 核心接口

- `resolveModel(provider: string, modelId: string, agentDir?: string, cfg?: Config)`

返回：

- `model?: Model`
- `error?: string`
- （可选）`authStorage`、`modelRegistry` 用于调试

## 3.2 获取 model 的标准流程

1. 解析 `agentDir`。
2. 创建认证存储（`discoverAuthStorage(agentDir)`）。
3. 创建模型注册表（`discoverModels(authStorage, agentDir)`）。
4. 先从 registry 查找：`modelRegistry.find(provider, modelId)`。
5. 命中则返回标准化后的 model。
6. 未命中则尝试 inline providers（来自配置中的 providers.models）。
7. 再未命中则走受控 fallback（仅当 provider 已配置或 mock 场景）。
8. 全部失败返回 `error: Unknown model: provider/modelId`。

## 3.3 为什么不是 if-else 堆满 server

这里采用“集中式模型解析器”思路，而不是在调用方写很多 `if (provider===...)`：

- 调用方只依赖 `resolveModel(...)` 一个入口。
- provider 扩展只改模型层，不污染业务层。
- 错误可观测性更好（统一 error 格式和调试信息）。

这本质是“策略注册 + 统一入口”，接近策略模式的工程化落地。

## 4. 配置优先级与读取路径

当前实现增加了全局用户配置文件：

- 默认路径：`~/.fgbg/fgbg.json`
- 可通过环境变量覆盖：`FGBG_CONFIG_PATH`

读取顺序（同字段覆盖优先级）为：

1. 环境变量（最高优先，例如 `MINIMAX_API_KEY`）
2. 项目配置 `config/model.json`
3. 全局配置 `fgbg.json`
4. 代码内默认值（兜底）

说明：`model-config.ts` 会先合并“全局 + 项目”，然后 API Key 解析时再由环境变量覆盖。

## 4.1 配置优先级（建议）

可采用以下优先级：

1. 运行时环境变量（最高优先）
2. 本地认证存储（profile/keychain/file）
3. 项目配置文件（默认值与结构约束）
4. 内置 provider 默认值（兜底）

说明：优先级是“值解析优先级”，不等同于“provider 是否可用”。可用性还受 `models[]` 是否存在、auth 是否完整、discover 是否成功影响。

## 5. 最小调用链（到拿到 model 为止）

```ts
const agentDir = resolveAgentDir();
await ensureModelJson(); // 准备 providers/models
const result = resolveModel("minimax", "MiniMax-M2.1", agentDir, cfg);

if (!result.model) {
  throw new Error(result.error ?? "model resolve failed");
}

// 到这里，拿到了可用于后续推理的 model 对象
const model = result.model;
```

## 6. 本项目裁剪建议

当前版本已从“单 provider 最小闭环”升级为“多 provider 隐式解析 + 统一解析入口”。

建议保留：

- 多 provider builder（至少保持当前 6 个）
- `resolveImplicitProviders` 的环境变量自动发现
- `ensureModelJson` 的 merge + 幂等写入
- `resolveModel` 的 registry 命中 + inline fallback

可以先不做：

- 多 provider OAuth 流程
- Bedrock/Copilot 这类特殊鉴权分支
- 复杂 alias/allowlist 规则





# 整体项目设计

本节定义重构后的目标结构，用于把当前臃肿的 `server.ts` 拆分为“Web 层 + Agent 运行层 + Pi 执行层”。

## 目标调用链

前端 -> `src/middleware/web-layer.ts` -> `src/agent/run.ts` -> `src/agent/pi-embedded-runner/attempt.ts`

## 分层职责

### 1) `src/server.ts`（应用装配层）

职责只保留：

- 初始化 Express、CORS、静态资源。
- 注册路由（把 `/api/chat` 等路由挂到 web layer）。
- 启动端口监听与基础日志。

不再承载：

- 模型选择细节。
- Agent 事件流解析。
- SSE 回包拼装细节。

### 2) `src/middleware/web-layer.ts`（Web 中间层）

职责：

- 接收 HTTP 请求并做参数校验。
- 建立 SSE 响应头、维护连接生命周期。
- 调用 `run.ts` 暴露的方法，把流式事件转发给前端。
- 将错误统一转换为 HTTP/SSE 错误格式。

建议对外接口：

- `createWebLayer()`：返回 Express Router，注册 `/api/chat`、`/api/history`、`/api/clear`。

### 3) `src/agent/run.ts`（Agent 运行编排层）

职责：

- 负责“每次对话请求”的统一流程编排。
- 管理模型发现与选型（复用 `selectModelForRuntime`、`ensureModelJson`）。
- 管理会话读写（读取历史、落盘、清理）。
- 收集并转发流式事件给上层回调（不直接依赖 Express）。

建议对外接口：

- `getReplyFromAgent(params)`：执行一次会话并流式返回事件。
- `getHistory()`：读取会话历史。
- `clearHistory()`：清理会话历史。
- `logRuntimePaths()`：打印全局配置和会话路径。

说明：

- 该层采用 FP 思路：仅导出函数，不创建长生命周期 runtime 对象。
- `getReplyFromAgent` 返回的是标准化事件流，不是 HTML/SSE 字符串。
- 该层可记录请求时间线（request start、LLM 首包、request end）。

### 4) `src/agent/pi-embedded-runner/attempt.ts`（Pi 执行层）

职责：

- 仅负责调用 pi-core/pi-agent-core 的执行逻辑。
- 把 Pi 原始事件映射成项目内部统一事件。
- 通过回调逐条上报流式增量，不关心 Web 协议。

建议对外接口：

- `createAgentRuntime(params)`：根据入参创建单次请求使用的轻量 Agent。
- `runEmbeddedPiAgent(params)`：执行一次 prompt，流式回调事件。

说明：

- `createAgentRuntime` 在每次请求时创建，不在进程全局缓存。
- 该层不处理路由、不处理 SSE、不处理 HTTP 状态码。

## 会话与模型初始化时机

在应用启动阶段（`server.ts`）：

- 打印运行路径：`logRuntimePaths()`

在请求阶段（`run.ts`）：

- 每次请求都执行一次 `selectModelForRuntime()` 选型。
- 读取会话历史后，调用 `attempt.ts` 的 `createAgentRuntime(params)` 创建轻量 Agent。
- 调用 `runEmbeddedPiAgent()` 执行并流式回调，结束后落盘会话。

会话文件落盘目录约定：

- `src/agent/session/*`

## 流式事件标准（项目内）

建议统一以下事件类型，web-layer 只做透传：

- `message_start`
- `message_update`（包含 `delta`）
- `message_end`
- `done`
- `error`

这样上层前端不需要关心 Pi 事件细节，只消费统一协议。

## 为什么这样拆

- `server.ts` 变薄：只负责装配和启动。
- 可测试性提升：`run.ts` 与 `attempt.ts` 可单测，不依赖 Express。
- 可替换性提升：后续若替换 pi-core，只需改 `attempt.ts`。
- 可观测性提升：请求耗时、首包耗时集中在运行层记录。
